## 1. Streamlining Documentation Files

### Problem: README.md doubles as a CLI reference
The README currently contains detailed CLI flag documentation for every command. This makes it long (~240 lines) and hard to maintain as a "marketing + getting started" doc. The CLI reference should live in `docs/`.

**Actions:**
- **Trim README.md Commands section** to just a brief overview with 1-2 examples per command, linking to `docs/cli-reference.md` for full details.
- **Create `docs/cli-reference.md`** as the authoritative CLI reference (moved from README), covering every command, subcommand, flag, and mode.

### Problem: docs/OVERVIEW.md is partially stale
It describes the old `v0` command structure and references `bindfs`/`macFUSE` as the primary mechanism, but the main commands (`materialize`, `sandbox`, `volume`, `auth`) now run Docker-based. The `v0` subcommand tree is hidden.

**Actions:**
- **Rewrite docs/OVERVIEW.md** to reflect the current architecture: Docker-backed sandboxes as the primary execution model, with the `v0` bindfs-based commands noted as legacy/hidden.
- Update the Architecture section to match the actual package layout (add `agentconfig`, `auth`, `config` packages; update `sandbox` package description to include Docker, volumes, presets, image resolution).

### Problem: CONTRIBUTING.md is too thin
It lacks Docker as a prerequisite, doesn't mention `bin/amika` wrapper, doesn't describe the project structure, and says `make build` runs `go build ./...` (should be `go build -o dist/amika ./cmd/amika`).

**Actions:**
- **Expand CONTRIBUTING.md** with:
  - Docker as a prerequisite alongside Go 1.21+
  - macOS as the currently supported platform
  - Correct build command description
  - Project directory layout overview
  - How to run the smoke test plan (link to `docs/development/testing.md`)
  - How presets work (auto-built on first use)

## 2. Updates to AGENTS.md

### Problem: Code Structure section is incomplete
It only lists `cmd/amika/main.go`, `dist/`, and `bin/amika`. It doesn't describe the internal packages or the CLI command files.

**Actions:**
- Expand the Code Structure section to include:
  - `cmd/amika/sandbox.go` — Sandbox CRUD commands
  - `cmd/amika/materialize.go` — Docker-based materialize command
  - `cmd/amika/volume.go` — Volume management commands
  - `cmd/amika/auth.go` — Auth credential extraction
  - `cmd/amika/v0.go` — Legacy v0 mount/unmount/materialize (hidden)
  - `internal/sandbox/` — Docker sandbox management, presets, volumes, stores
  - `internal/auth/` — Credential discovery across Claude, Codex, OpenCode, Amp
  - `internal/agentconfig/` — Auto-mount agent credential files into containers
  - `internal/config/` — XDG path resolution, state file locations
  - `internal/mount/` — bindfs-based mount/unmount (v0)
  - `internal/materialize/` — Local sandbox materialize (v0)
  - `internal/state/` — JSONL mount state persistence (v0)
  - `internal/basedir/` — XDG base directory resolution

### Problem: Missing guidance on Docker dependency
AI agents need to know Docker is required for most commands and how preset images work.

**Actions:**
- Add a "Runtime Dependencies" section noting Docker is required for `materialize`, `sandbox`, and `volume` commands
- Add note about preset images auto-building on first use
- Add note about `AMIKA_STATE_DIRECTORY` env var override

### Problem: Missing guidance on test patterns
AI agents should know about `AMIKA_RUN_EXPENSIVE_TESTS` and `GIT_CONFIG_GLOBAL=/dev/null` for reliable test runs.

**Actions:**
- Add a "Testing Notes" subsection with these caveats
